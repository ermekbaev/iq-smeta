# IQ SMETA — рабочий статус и шпаргалка

> Живой документ для самопроверки: что готово, что осталось, что нужно от заказчика,
> как запускать, куда что развивать. Обновляй по ходу.
> Бюджет MVP по ТЗ: **40 000 ₽**, через Кворк, 2 этапа, срок «ориентировочно 2–3 недели».

---

## 1. Коротко: что это и на чём

Голосовое создание смет: надиктовал материалы/работы → распознавание речи →
ИИ извлекает позиции → умный подбор по прайсу → фирменная смета в PDF. PWA (ставится на телефон).

**Стек:** Next.js 16 + React 19, PostgreSQL + pgvector, Prisma 7, NextAuth v5,
Puppeteer (PDF), Tailwind 4. ИИ — через адаптер (prod-курс: Яндекс).

---

## 2. Как запустить локально

```bash
# 1. База (Docker) — Postgres+pgvector, порт 5433 (5432 занят системным PG)
docker compose up -d

# 2. Миграции + сид админа (один раз)
npx prisma migrate deploy
npx prisma db seed

# 3. Запуск
npm run dev            # http://localhost:3001
npm test               # юнит-тесты (vitest)
npm run build          # прод-сборка
```

**Вход:** `admin@iqsmeta.local` / `admin123` (поменять на проде!).
**Экраны:** `/login`, `/admin` (панель), `/admin/price` (прайс), `/admin/estimates` (сметы), `/record` (запись/сборка сметы).

**.env** (не коммитить): `DATABASE_URL`, `AUTH_SECRET`, `AI_PROVIDER=prod`,
`ASR_PROVIDER=gemini`, `LLM_PROVIDER=gemini`, `GEMINI_API_KEY=...`,
`COMPANY_NAME/COMPANY_CONTACTS/COMPANY_LOGO_URL` (реквизиты в PDF).

## 4. Что осталось под уверенную сдачу 40к

Это **не новые фичи**, а доводка — и упирается в материалы заказчика:

1. **Настройка извлечения/подбора на его реальных надиктовках** (пункт ТЗ «извлечение позиций и количеств»). Механизм готов, точность на его манере речи докрутить → **нужны 5–10 примеров надиктовок**.
2. **Реквизиты/логотип в PDF** → по данным заказчика (сейчас заглушка в шапке).
3. **UX добавления позиции вне прайса** — проверить, что «такого нет → завожу вручную с ценой» удобно (сейчас есть «Ввести вручную»).
4. **Деплой** на хостинг + боевой `AUTH_SECRET` (см. раздел 7).

## 6. Технические факты и грабли (важно помнить)

- **Оплата ИИ из РФ** — Gemini/OpenAI/Claude из России картой не оплатить. Для прода: либо заказчик платит доступный ему сервис, либо российские GigaChat/Яндекс. Разработчик из КР российские тоже не оформит — по ТЗ это зона заказчика. Провайдер меняется в `src/lib/ai/index.ts` одной заменой.
- **Лимит бесплатного Gemini — 100 эмбеддингов/мин.** Большой прайс грузится медленно (215 поз ≈ 130 с): батч 90 + ретраи, которые читают из 429 «сколько ждать». На платном тарифе лимита нет. → Для прода заказчику нужен платный аккаунт.
- **ИИ нужен в 3 местах на смету:** ASR (голос→текст), LLM (текст→позиции), эмбеддинг (название→вектор для поиска). Сам подбор и сборка сметы — чистая база/код, без ИИ. Эмбеддинги прайса считаются один раз при загрузке.
- **Размерность вектора** зашита под схему БД: сейчас **256** (`vector(256)`, `EMBEDDING_DIM=256`). Gemini запрашивается с `outputDimensionality=EMBEDDING_DIM`; при переходе на провайдера без такой настройки (например GigaChat 1024) — сменить `EMBEDDING_DIM` в `src/lib/ai/types.ts` + `vector(N)` в схеме + миграция + **перезалить прайс** (старые векторы из другой модели не годятся). Guard в index.ts кинет понятную ошибку при несовпадении.
- **GigaChat TLS** — нужен «Russian Trusted CA» через `NODE_EXTRA_CA_CERTS` (иначе запросы падают).
- **Порт БД 5433** (не 5432 — занят системным PostgreSQL).
- **Модель категорий** — свободный текст (реальные разделы прайса), не фиксированный список.
- **Цена в смету — РРЦ** (розница). ОПТ (закупка) хранится в `cost` под будущую маржу.

## 7. Деплой (когда дойдём)

- **VPS:** Ubuntu 24, домена нет.
- **HTTPS обязателен** (микрофон и установка PWA работают только по https).
  - Без домена: бесплатный поддомен **DuckDNS** + **Caddy** (авто-сертификат), либо `cloudflared` туннель (быстро, но адрес временный).
- **Стек в Docker Compose:** app (Next.js) + db (pgvector) + Caddy (авто-HTTPS).
- **Подводные камни:** pgvector (нужен образ `pgvector/pgvector`), Puppeteer (Chromium — память/зависимости; на 1 ГБ RAM может падать).
- **Рекомендация:** перевести PDF с Puppeteer на **`@react-pdf`** (без Chromium) → лёгкий деплой на любой хостинг, меньше памяти. ~1 час работы.
- Боевой **`AUTH_SECRET`** (`openssl rand -base64 32`) и смена пароля админа.

## 8. Видение заказчика → дорожная карта (ФАЗА 2+, вне 40к)

Заказчик прислал голосом видение «умной сметы-помощника». Почти всё это — **за рамками 40к** (он сам говорит «потом»). Оформляем как платные очереди развития (путь к 100к+):

- **Модуль «умная сборка сметы»:**
  - **Цена/работа голосом** («монтаж 20 тысяч») — расширить извлечение LLM.
  - **Узлы/комплекты (BOM):** одна фраза → набор позиций. Рекомендуемая модель — общая «расценка/состав изделия» (как технологическая карта ГЭСН/ФЕР): позиция = товар | работа | узел; у узла состав `{позиция, кол-во}`. Покрывает и «работа привязана к товару», и «монтаж узла дождевателя».
  - **Позиции вне прайса голосом** («нестандартные работы 20к»).
- **Модуль финансов:** себестоимость/маржа, вопрос ИИ «какая прибыль?». Фундамент заложен (поле `cost`).
- **Интеграции:** 1С, склад, наличие.
- **ИИ-помощник:** самообучение, анализ, подсказки (всегда с ручной проверкой).
- Из ТЗ отдельно (фаза 2): Telegram-бот, парсер прайсов из PDF, экспорт в DOCX.

**Вопросы к заказчику для будущего дизайна:** узлы 1:1 или комплекты? откуда связка «товар↔работа»? как часто реально «цена голосом»?

## 9. Карта проекта (где что лежит)

```
src/
  app/
    (admin)/            # админка (layout с таб-баром)
      admin/price/      # управление прайсом (PriceManager)
      admin/estimates/  # список + деталь + редактирование сметы
    record/             # экран записи → сборка сметы (главный UX)
    api/
      asr/              # аудио → текст
      extract/          # текст → позиции (LLM)
      match/            # подбор по базе
      estimate/         # CRUD смет + /pdf
      admin/price/      # загрузка/CRUD прайса
      auth/             # NextAuth
  lib/
    ai/                 # адаптеры: index(выбор), gemini, gigachat, yandex, mock, util(ретраи/батч)
    match/              # matcher (alias→pgvector→нечёткое→порог), fuzzy
    price/              # parse (разбор Excel/CSV), service (импорт+эмбеддинги)
    estimate/           # service (сборка/группировка/маржа-хелпер)
    pdf/                # template (HTML) + render (Puppeteer)
    audio/              # wav-recorder (запись в браузере)
  components/BottomNav.tsx
prisma/                 # schema + migrations + seed
СТАТУС.md  PLAN.md  README.md  AGENTS.md
```

**Тюнинг подбора (env, без правок кода):** `MATCH_THRESHOLD` (порог, 0.6), `MATCH_W_SEMANTIC` (0.7), `MATCH_W_FUZZY` (0.3).

## 10. Деловые заметки по сдаче

- Раньше срока — **нормально** (в ТЗ «ориентировочно 2–3 недели»). Заказчик платит за результат по этапам, не за часы. **Не говорить «сделал за день»** — не усиливает позицию.
- Сдавать **по этапам** ТЗ (Этап 1 → приёмка → оплата → Этап 2), лучше уже на его данных.
- Чётко разделять: **«это MVP по ТЗ»** (готово) vs **«это дальнейшее развитие»** (узлы, цена голосом, маржа, 1С, самообучение — он сам назвал «потом»).
- **Авторизация** уже в цене MVP (блок «архитектура+авторизация 5000₽») — отдельно не выставлять. Доп. оплата уместна только за то, что *сверх* базовой (роли, кабинеты, восстановление пароля и т.п.).
- 2 недели поддержки после сдачи — по ТЗ; закладывать время на правки при реальном использовании.
