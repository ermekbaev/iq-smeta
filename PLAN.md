# IQ SMETA — план разработки MVP

Голосовое создание смет: пользователь надиктовывает материалы и работы → система
распознаёт речь, извлекает позиции, подбирает их по базе прайсов и формирует
фирменную смету в PDF. Работает как сайт и устанавливается на телефон (PWA).

Бюджет MVP: 40 000 ₽, через Кворк, двумя этапами. Срок ~2–3 недели.

---

## 1. Объём MVP

**Входит:**
- PWA с установкой на телефон (Android/iOS через браузер)
- Админка: загрузка прайса (Excel/CSV) + ручное редактирование позиций
- Голосовой ввод: запись + расшифровка речи
- ИИ-извлечение позиций и количеств из надиктованного
- Умный подбор позиций по базе (находит «цемент М500», даже если в прайсе он записан иначе)
- Сборка сметы + генерация фирменного PDF
- Базовая авторизация
- Простая фиксация правок (запоминание сопоставлений)

**НЕ входит (фаза 2, отдельно):**
- Telegram-бот (переиспользует движок, добавляется быстро)
- Парсер прайсов из PDF (на старте только Excel/CSV)
- Экспорт в DOCX
- Полноценное самообучение ИИ

## 2. Архитектура и стек

- **Фронтенд / PWA:** Next.js (React), manifest + service worker для установки и офлайн-оболочки
- **Бэкенд:** Node.js (можно как API-роуты Next.js, чтобы не плодить сервисы)
- **База:** PostgreSQL, расширение **pgvector** для семантического подбора
- **ORM:** Prisma
- **Распознавание речи (ASR):** Яндекс SpeechKit (рублёвый биллинг, РФ)
- **Нейросеть (LLM, извлечение позиций):** YandexGPT или GigaChat (рублёвый биллинг, РФ)
- **Эмбеддинги (для подбора по базе):** YandexGPT Embeddings или GigaChat Embeddings (рублёвый биллинг, РФ). Размерность вектора фиксируется под выбранного провайдера ДО первой миграции БД — поменять потом дорого (см. 4).
- **Генерация PDF:** HTML-шаблон → Puppeteer (фирменный вид с логотипом и таблицами). На дешёвом хостинге Puppeteer тянет полный Chromium (память + зависимости) — заложить VPS с запасом по RAM либо рассмотреть `@react-pdf`/serverless-Chromium как лёгкую альтернативу (см. 8).

Поток данных:
`Голос → ASR (текст) → LLM (структура позиций) → подбор по базе (pgvector + нечёткий) → сборка сметы → PDF`

Аудио после расшифровки **не храним** (отправили в ASR → получили текст → удалили). Если понадобится хранение для разбора ошибок — добавляем явно и осознанно.

Задержка не критична (цель 1–2 минуты на всю цепочку), это диктовка, а не диалог —
поэтому Vapi и реалтайм-голос НЕ нужны.

## 3. Ключевые технические решения и нюансы

### 3.1. Провайдеры ИИ — под оплату из РФ
- Claude/OpenAI/Whisper из России официально не оплатить (карты не проходят).
- Поэтому в продакшене: **Яндекс SpeechKit (речь) + YandexGPT/GigaChat (извлечение)** — рубли, РФ.
- Аккаунты этих сервисов оформляет **заказчик** под своим биллингом, настройка с нашей помощью.
- LLM, ASR **и эмбеддинги** вынести в **адаптер/интерфейс**, чтобы провайдер менялся одной заменой, без переписывания логики. На разработке можно гонять на чём удобно, прод — на российских.
- **ASR-нюанс:** диктовка прораба может быть длинной и шумной. Уточнить лимит длины короткого распознавания SpeechKit и при необходимости использовать долгий/потоковый режим, а не sync на короткие фразы.

### 3.2. Умный подбор по базе (сердце точности)
Голос не совпадает с прайсом дословно. Подбор в два слоя:
1. **Семантический поиск** через pgvector: эмбеддинг названия позиции из речи ищет ближайшие по смыслу записи прайса.
2. **Нечёткое сравнение** (по строке) как дополнительный сигнал.
3. Если уверенность низкая — показываем варианты на подтверждение, а не молча угадываем.

**Проверка качества (обязательно до приёмки этапа 2):** собрать набор из 10–20 реальных фраз прораба с эталонными позициями прайса и мерить долю корректных автоподборов. Без этого подбор нельзя ни настроить, ни принять. На общих эмбеддингах синонимы стройтерминов («цемент М500» ↔ «ЦЕМ I 42,5Н») могут не сводиться — за это отвечает слой aliases (3.3) и порог уверенности.

### 3.3. Фиксация правок («обучаемость» MVP)
- Таблица **aliases**: `сказанный_текст → id_позиции_прайса`.
- Админ поправил сопоставление один раз → в следующий раз подбор сразу берёт правильную позицию.
- При конфликте (одна фраза → разные позиции в разных правках) для MVP принимаем правило **«последняя правка побеждает»** (уникальность по `spoken_text`).
- Это простая, но рабочая версия «обучения». Полное самообучение — фаза 2.

### 3.4. Извлечение позиций (LLM)
- На вход: расшифрованный текст. На выход: список `{наименование, количество, единица}`.
- Использовать структурированный вывод (строгий JSON), чтобы не парсить свободный текст.
- Пример: «10 мешков цемента М500, 3 куба песка и доставка самосвалом» →
  `[{наименование:"цемент М500", кол-во:10, ед:"мешок"}, {наименование:"песок", кол-во:3, ед:"м3"}, {наименование:"доставка самосвалом", кол-во:1, ед:"шт"}]`
- **Нормализация единиц:** единица из речи («мешок», «куб», «тонна») может не совпадать с единицей прайса («шт», «м3», «т»). Завести минимальный словарь синонимов единиц и приводить к канону при сборке сметы — иначе количество/итог считаются неверно.

### 3.4a. Обработка ошибок цепочки
Голосовой UX наполовину состоит из понятных ошибок. Для каждого шага — внятное состояние:
- ASR ничего не распознал / тишина → «не расслышали, повторите».
- LLM вернул пустой/невалидный JSON → показать сырой текст и дать поправить вручную.
- Подбор пустой или низкая уверенность → показать варианты на выбор (см. 3.2), не подставлять молча.

### 3.5. Генерация PDF
- Шаблон в HTML/CSS (логотип, реквизиты, таблица позиций, итог), рендер через Puppeteer.
- Разбивка по категориям: Материалы, Работы, Оборудование, Доставка, Издержки.

### 3.6. PWA
- manifest.json + service worker, чтобы браузер предлагал «Установить приложение».
- Базовый офлайн-shell; запись голоса через Web Audio API.
- **Граница офлайна:** офлайн работает только оболочка приложения. Расшифровка, извлечение и подбор требуют сети — это онлайн-обработка. Зафиксировать ожидание с заказчиком, чтобы не ждал полностью офлайн-работу.

## 4. Модель данных (черновик)

- **users** — id, phone/email, password_hash, role (admin)
- **price_items** — id, name, unit, price, category, embedding (`vector(N)` — N фиксируется под embeddings-провайдера, см. 2)
- **aliases** — id, spoken_text (уникально), price_item_id
- **estimates** — id, user_id (автор), title, client_name, created_at, total
- **estimate_items** — id, estimate_id, price_item_id?, name, qty, unit, price, sum

Категории позиций: Материалы, Работы, Оборудование, Доставка, Издержки.

**Границы MVP по пользователям:** один тип пользователя (admin), без регистрации клиентов и без многопользовательского разграничения. Сметы создаёт тот же admin. Это зафиксировано, чтобы не разойтись в ожиданиях на приёмке.

## 5. Структура проекта (ориентир)

```
iq-smeta/
  app/                 # Next.js (PWA фронт + API-роуты)
    (admin)/           # админка: прайс, редактор, сметы
    record/            # экран записи голоса
    api/
      asr/             # приём аудио → текст (Яндекс SpeechKit)
      extract/         # текст → позиции (LLM-адаптер)
      match/           # подбор по базе (pgvector)
      estimate/        # сборка сметы
      pdf/             # генерация PDF
  lib/
    ai/                # адаптеры ASR и LLM (сменяемые провайдеры)
    pdf/               # HTML-шаблон + Puppeteer
  prisma/
    schema.prisma
  public/
    manifest.json
```

## 6. Этапы разработки (под оплату на Кворке)

### Этап 1 — фундамент + ввод данных
- Каркас проекта (Next.js + Prisma + PostgreSQL + pgvector)
- Базовая авторизация
- Админка: загрузка прайса Excel/CSV, разбор файла по колонкам (наименование, ед., цена, категория), ручное редактирование позиций
- Экран записи голоса + расшифровка через ASR (текст на экране)
- **Результат этапа:** прайс грузится и редактируется, голос записывается и превращается в текст.
- **Критерии приёмки:** загрузка реального файла на N строк → ровно N позиций в базе с корректными колонками; правка позиции сохраняется; запись голоса даёт читаемый текст на экране.

### Этап 2 — ИИ, подбор, документ, PWA
> **Блокер до старта этапа 2:** на руках реальный прайс заказчика + набор из 10–20 надиктовок с эталонными позициями (см. 7). Без них подбор нельзя настроить и принять.

- LLM-извлечение позиций и количеств (структурированный JSON)
- Умный подбор по базе (pgvector + нечёткое сравнение) + таблица aliases
- Сборка сметы (позиции, цены, количество, итог)
- Генерация фирменного PDF
- Сборка PWA (manifest, service worker, установка на телефон)
- **Результат этапа:** надиктовал → получил готовую смету в PDF, приложение ставится на телефон.
- **Критерии приёмки:** на тест-наборе надиктовок доля корректных автоподборов ≥ согласованного порога (см. 3.2); готовый PDF с фирменным оформлением и разбивкой по категориям открывается на телефоне; приложение ставится на Android/iOS через браузер.

## 7. Что нужно от заказчика
- Прайс-лист в Excel/CSV (с колонками: наименование, ед. изм., цена, категория)
- Фирменное оформление для PDF: логотип, контакты/реквизиты, желаемый вид таблицы
- Аккаунты Яндекс SpeechKit и YandexGPT/GigaChat под его биллинг (настроим вместе)
- Примеры реальных надиктовок (как реально говорят прорабы), 10–20 шт. с эталонными позициями прайса — для настройки и приёмки подбора (блокер этапа 2)

---

## 8. Расходы на эксплуатацию (на стороне заказчика, после запуска)
- Распознавание речи + нейросеть + эмбеддинги: ~1–3 ₽ за смету, ~500–2000 ₽/мес при умеренной нагрузке
- Хостинг + база: ~1000–2500 ₽/мес. **Оговорка:** Puppeteer требует памяти под Chromium — на самом минимальном тарифе генерация PDF может падать. Либо берём VPS с запасом по RAM, либо переходим на лёгкий рендер PDF (`@react-pdf`/serverless-Chromium).
- Домен: ~200–1000 ₽/год

## 9. Открытые вопросы (уточнить с заказчиком)
- Точная структура его прайса (колонки, единицы, есть ли категории)

- Нужна ли в смете шапка с данными клиента/объекта
- Где хостим (его VPS, облако) и хватает ли RAM под Puppeteer
- Целевой порог качества автоподбора для приёмки этапа 2
